Introduction


Hi and welcome to the documentation for the astropy API. First of all, thank you for visiting this page. If you liked to use the API to get stars and constellations information don’t forget to sign up, confirm your email address and create and API key.

This is my first attempt at creating a fully functional API and I honestly hope it could help you develop your project, explore the sky, or get to know a bit more about astronomy.

Currently this API provides information about stars and constellations, however as the project progresses planets will be included, too. Oh! and don’t forget about the Moon, and the other moons of course.

Below you can see the endpoint for this API, and various example of requests and response, as well as general instructions to make proper requests and a bit of insights about the information provided in the response.



Requests


All routes accept GET requests only and they require certain parameters, that are specified in this documentation, depending on the specific route. One parameter that is always needed is “appid” which represents the API key associated to your account. At the moment, all the API keys are created equal, there is no special privilege, or super user, etc. You will see a couple of admin-only route and they are specifically there with an eye on the future, when the keys might get special permissions.

As per standard all the keys enable you to make requests to all the endpoints, excluded the admin-only ones, to a maximum of 1000 calls a day. I understand it is not much, but once you reach the call limit you have to wait for the day to end. As I previously mentioned, this project has eyes on the future, I want to expand and bring it forward as far as I can, and I can assure you that there will be opportunities for API keys with unlimited access and number of calls. As the zen master says…just Be Patient.



Response


All the responses are returned in JSON format, there might be an XML version in the future if the users want it. By the way, if you wanted to leave comments about the API, suggestions, feedbacks or even requests that you feel would improve it, please don’t hesitate to hit the Feedback page, I am always open to collaborations. In addition, this project is on Github, so you can make pull requests, etc. I’d be happy to have you on board.

Along with the JSON response, you receive a status code depending on the outcome of your request. If you are not familiar with them, you can visit this page on Wikipedia that explains them in detail. However, for this API the range of status codes is very narrow. There is 200 OK (of course), for successful requests, 204 No Content, 400 Bad Request, 403 Forbidden, 404 Not Found. In the documentation, it is explained when these codes might arise.

Now, let’s proceed with 2 routes that have public access and no api key is required. They are only meant to see whether or not the API is functioning properly.



Home route


/astropy

This route will help you assess wheter the API is working correctly or not. It will return a simple JSON file with a Welcome message and a 200 OK status code. Like this:


{ "Welcome": "This is AstroPy" }


Greeting message


/astropy/hello

This route accepts only GET requests with two simple arguments: name and last.

As the name suggests it will return a JSON response with greetings to name and last. It is a simple way to test arguments reception by the API. The JSON response will have a 200 status code if everything was sent correctly, a 204 if no argument is passed in, or a 400 if a bad request was submitted, or a wrong parameter was passed in or not enough parameters. This errors will be present for all the requests that do not fulfil the requirements. So, 200 OK for good requests, 204 if no argument is passed in and a 400 for bad requests, in addition 404 status code in case the request was correctly submitted but no record was found, or the URL was not correct.


Request/Response examples:


In case of 200 OK:


GET request: /astropy/hello?name=BeGeos&last=Dev


Response: { "Hello, There": "BeGeos Dev" }


In case of 204 No Content:


GET request: /astropy/hello


Response: No content will be shown


In case of 400 Bad Request:


GET request: /astropy/hello?nme=BeGeos


Response: { "message": "Bad Request" }


Be careful of reading through the documentation to check all the required parameters and the correct endpoints so that your requests will always yield a positive response.



API key


Every route, apart from the first 2 presented, needs "appid" among the required arguments. If an appid is not provided the response will return a "Missing Token" message and a 400 Bad Request status code.


{ "message": "Missing token" }

If invalid or expired appid is provided the response will return "Invalid Token" message and a 403 Forbidden status code.


{ "message": "Invalid token" }

To generate an api key simply register to this website here and confirm your email address. Api keys are fundamental to make requests to this API.

In case you were already a confirmed user, you can generate an API key at your profile page or clicking here if you are already logged in.

Constellations


As far as constellations are concerned 3 routes are provided with different endpoints and parameters.

The first one is a basic GET request to return the information about a constellation, the second one is a more advanced query search that takes minimum and maximum latitude into account, as well as the quadrant. If you don't know what these are please refer to the 'How to Observe' page and you'll get some insights. The third endpoint is an exclusive admin only request to return all the constellations in the database. It will be updated for special accounts, but it is not provided yet, since all accounts have the same privileges.



Single Constellation


/astropy/api/v1/constellation

This route accepts only one parameter, namely c (as for constellation). Any other parameter won't throw an error, as long as c is in there, but it won't be considered.

As previously mentioned, this route will return all the information about the constellation provided in the c argument via a GET request.

If requests are sent correctly the response will be a 200 OK and the JSON response will look like this:


GET request: /astropy/api/v1/constellation?c=ursa%major&appid=YOUR_API_KEY


Response:


                

                

                

                  The argument is not case sensitive since each one will be
                  converted into lowercase in the process. 


                  As you can see the request was sent correctly and the response
                  yielded the correct result. The main key is "constellation"
                  which contains the information.
                

                

                
Elements Explained:

                

                

                  

                    constellation.abbreviation
                  

                  

                    

                      It provides the abbreviation code, as per IAU standards,
                      of the constellation;
                    

                  

                  
constellation.alias

                  

                    

                      It represents the most common translation, or meaning
                      given to the name of said constellation;
                    

                  

                  

                    constellation.declination
                  

                  

                    

                      As the name implies it represents the average declination
                      of the constellation. If you wanted to know more about
                      declination/right ascension, please visit the dedicated
                      page 'How to Observe';
                    

                  

                  
constellation.id

                  

                    

                      It shows the id of that particular constellation. It
                      doesn't have any observational relevancy;
                    

                  

                  

                    constellation.max_latitude
                  

                  

                    

                      It provides the maximum latitude where the constellation
                      is visible. As the definition implies, maximum means that
                      beyond that latitude the constellation is not observable;
                    

                  

                  

                    constellation.min_latitude
                  

                  

                    

                      It provides the minimum latitude where the constellation
                      is visible. As the definition implies, minimum means that
                      the constellation is only observable above that latitude;
                    

                  

                  
constellation.name

                  

                    

                      Self-explanatory. It returns the standard name of the
                      constellation, in lower case;
                    

                  

                  

                    constellation.quadrant
                  

                  

                    

                      It returns the quadrant where the constellation is placed
                      in the celestial sphere. As a general rule quadrant
                      starting with N, indicates the Northern Hemisphere. While
                      S points to the Southern Hemisphere;
                    

                  

                  

                    constellation.right_ascension
                  

                  

                    

                      It provides the average right ascension of the
                      constellation. If you wanted to know more about
                      declination/right ascension, please visit the dedicated
                      page 'How to Observe';
                    

                  

                  
constellation.stars

                  

                    

                      It returns the main stars in the constellation. Each star
                      has a name and apparent magnitude argument;
                    

                  

                  

                    constellation.stars.apparent_magnitude
                  

                  

                    

                      It indicates the brightness of the star. Be aware, the
                      result is not ordered, and more importantly, the lower the
                      magnitude the higher the brightness;
                    

                  

                  

                    constellation.stars.name
                  

                  

                    
It gives the name of the star in lower case.

                  

                

                

                

                

                  If requests were not correctly submitted, either missing or
                  misspelled argument the response will give a message and a 400
                  Bad Request error:
                

                

                

                  GET request:
                  /astropy/api/v1/constellation?C=ursa%major&appid=YOUR_API_KEY
                

                

                
Response: { "message": "400 Bad request" } 

                


                

                  In case a request does not have any arguments the response
                  will return a 400 Bad Request status code and a message:
                

                

                

                  GET request:
                  /astropy/api/v1/constellation?appid=YOUR_API_KEY
                

                

                

                  Response: { "message": "No argument passed in" } 
                

                


                

                  If the name of the constellation were wrong or non-existent,
                  you'll simply get a 404 error and no response.
                

              


              


              

              

                
Query Based Search

                

                /astropy/api/v1/query
                


                

                  This route is meant to have a more complex logic behind it. It
                  represents a query to the database. It allows for a bit of
                  flexibility as far as the parameters are concerned. There are
                  only 3 key parameters all the others won't throw any error but
                  they won't be considered.
                

                

                

                  The query can be based upon the quadrant, as "q" parameter, in
                  this way:
                

                

                

                  GET request:
                  /astropy/api/v1/query?q=NQ2&appid=YOUR_API_KEY
                

                

                

                  A JSON response with a 200 status code and all the
                  constellations that fits the query parameter, namely are in
                  the NQ2 quadrant. 


                  Or it could be based on minimum latitude (as min) and/or
                  maximum (as max) latitude:
                

                

                

                  GET request:
                  /astropy/api/v1/query?min=-10&max=90&appid=YOUR_API_KEY
                

                

                

                  A JSON response with a 200 status code and the constellations
                  that satisfy the query parameters. If the query yields no
                  record, an empty response with a 404 code and a message "No
                  record found" will be delivered. Be aware that min and max are
                  integers and negative ones indicate southern latitudes, below
                  the Equator. 


                  You can also concatenate these queries creating a more detail
                  search, providing q, min and max as parameters. The result
                  will be a fit of all the conditions specified in the request.
                  


                  As always if no argument is passed in, the response will
                  return a 400 Bad Request and a "No argument passed in"
                  message.
                  


                  If arguments are provided but they are the wrong ones, instead
                  of q is a Q for instance, you'll receive a 404 not found code
                  and a "No record found" message. Only the correct arguments
                  will be considered by the API.
                

              

              


              

              

                
All Constellations (@admin only route)

                

                /astropy/api/v1/constellation/all
                


                

                  As of now this is an admin only route, which means that to
                  make a request to this endpoint you need to have admin
                  privileges, and an admin key. It only accepts POST requests,
                  and it will return a JSON response with all the
                  constellations, and their information. 


                  For future reference to make a request you need to provide in
                  the JSON body of the POST request, the “ADMIN_KEY” argument.
                  If none or incorrect argument are provided, you'll receive a
                  404 Not Found code and a "No valid argument" message.
                  


                  On the other hand, if an invalid ADMIN_KEY was provided, a 403
                  Forbidden and a "Invalid Admin Key" message will be received.
                  


                  As the name suggests this route will return a JSON response
                  with a list of all the constellations and their information.
                  FYI there are 88.
                

              

            

          

          

            
Stars

            

              

                In the case of stars there are 4 public routes and 1 admin only.
                


                As for the constellation one, api keys must be provided in all
                public requests.
                


                The first endpoint returns information about a star, the second
                one has a more complex logic behind it, and it will return
                instructions about how and where to look to observe said star,
                and it is based upon location. The third and fourth ones are
                fairly similar and they return a response of the closest or the
                brightest stars in the sky. The admin only route provides a
                complete list of all the stars in the database.
              

              


              

              

                
Single Star

                

                /astropy/api/v1/star


                

                  This route accepts only one parameter, all the others will not
                  be considered. The s parameter, for star, is required.
                  


                  A successful request looks like this:
                

                

                

                  GET request:
                  /astropy/api/v1/star?s=polaris&appid=YOUR_API_KEY
                

                

                
Response:

                

                  

                

                

                

                  The argument is not case sensitive; therefore, Polaris is
                  equal to polaris. 


                  In the response the main key is star, from there you can get
                  access to all the different information here explained:
                

                

                

                  

                    star.apparent_magnitude
                  

                  

                    

                      It indicates the brightness of a star, namely its
                      visibility in the night sky. It is important to keep in
                      mind that the lower the apparent magnitude the higher the
                      brightness. To have a more comprehensive description,
                      please refer to the "How to Observe" page;
                    

                  

                  
star.constellation

                  

                    

                      It provides the name and the abbreviation of the
                      constellation the star belongs to. To access it just
                      follow the star.constellation.name for the name, or
                      star.constellation.abbreviation for its abbreviation;
                    

                  

                  
star.declination

                  

                    

                      It provides the declination of the star which, along with
                      its right ascension, are the fundamental parameters for
                      astronomical observation. Make sure to visit the "How to
                      Observe" page to read a thorough description of both;
                    

                  

                  
star.distance

                  

                    

                      It indicates the distance in ly (lightyears) from Earth;
                    

                  

                  
star.name

                  

                    
It says the name of the star in lower case;

                  

                  
star.right_ascension

                  

                    

                      It provides the right ascension of the star which, along
                      with its declination, are the fundamental parameters for
                      astronomical observation. Make sure to visit the "How to
                      Observe" page to read a thorough description of both;
                    

                  

                  
star.type

                  

                    

                      It indicates the spectral type of the star. It suggests
                      the stars' colour, mass and whether it's a binary,
                      multiple or single star system.
                    

                  

                

                


                

                  If requests are not correctly submitted, either missing or
                  misspelled argument the response will give a message and a 400
                  Bad Request error, for instance:
                

                

                

                  GET request:
                  /astropy/api/v1/star?S=polaris&appid=YOUR_API_KEY
                

                

                
Response: { "message": "Bad Request" } 

                


                
or GET request: /astropy/api/v1/star

                

                

                  Response: { "message": "No argument passed in" } 
                

                

                

                  In case no record was found the response will give a 404 Not
                  Found error.
                

              

              


              

              

                
Where to Look

                

                /astropy/api/v1/where-to-look


                

                  This URL enables complex logic to calculate the current star's
                  position in the sky from a certain location. The response
                  might be different based upon the data received. It works with
                  2 required arguments. First: "s" (star) is fundamental since
                  it will take this star into consideration for the
                  calculations. Second, either the name of the city (as city) or
                  the latitude (lat) and longitude (lon) of a location.
                  


                  Always make sure to parse the arguments correctly or the
                  response will throw an error, in the case of latitude and
                  longitude they should start with numbers, so N67 is not valid,
                  and always have a + or - in front of the latitude. An example
                  of an acceptable format is lat = -87 lon = 180. Precise
                  coordinates are not necessary since the calculation will only
                  take the degrees into consideration. If you provide precise
                  coordinates the engine won't work correctly. The second option
                  is to provide the name of a city (it is not case sensitive).
                  The engine will use an API to get the coordinates and start
                  all the calculations.
                  


                  Request examples:
                

                

                

                  GET request:
                  /astropy/api/v1/where-to-look?s=polaris&lat=-10&lon=80&appid=YOUR_API_KEY
                

                

                
Response:

                

                  

                

                

                

                  As you can see the response gives some basic information about
                  the star but mainly the “where” key tells you where to look.
                  In this case, Polaris is not observable from latitude -10
                  because it is notoriously only visible in the Northern
                  Hemisphere.
                

                

                

                  GET request:
                  /astropy/api/v1/where-to-look?s=polaris&city=London&appid=YOUR_API_KEY
                

                

                
Response:

                

                  

                

                

                

                  This response refers to the current time when the request is
                  sent, and the hours are displayed in the time zone of the
                  location. Some of these items are well-know, but the main
                  things you want to look at is the star rise and set times,
                  which tells you when this star is going to rise and set at
                  said location. In this case, the "it rises" item doesn't
                  provide a time of day because Polaris is a circumpolar star
                  (see "How to Observe- Nomenclature") therefore it is always
                  above the horizon at this latitude. The sunrise and sunset
                  times are also very important because they tell you when you
                  can start observing the sky. In the "where" section you can
                  see instructions to follow to locate the star from this
                  location. Since Polaris never rises, nor set at this location,
                  there is no E-W movement, although since London's latitude is
                  below Polaris' declination you need to compensate that by
                  looking at the zenith plus 38° to the North. For a complete
                  guide refer to the How to Observe page in this website.
                

                


                

                  GET request:
                  /astropy/api/v1/where-to-look?s=hamal&city=London&appid=YOUR_API_KEY
                

                

                
Response:

                

                  

                

                

                

                  In this response the star Hamal was requested, in the Aries
                  constellation. Rise and set times are 11 AM and 23 (meaning 11
                  PM), thankfully all the stars apart from the Sun always remain
                  12 hours above the horizon. From London, coordinates are
                  specified in the “lat” and “lon” arguments, the current
                  position says that this star has already set, therefore is not
                  visible at the moment (8 AM in London). Just wait patiently
                  for 3 hours and Hamal should rise. Unfortunately, stars are
                  not visible in the Sky when the Sun is up, due to its
                  overwhelming brightness. So, if you want to observe this star
                  you have time until it sets at 23, knowing that the Sun sets
                  at 17.15 gives you almost 6 hours for observation. Although,
                  since the sunset doesn’t completely switch the lights off, you
                  should realistically have 3-4 hours for observation.
                  


                  One important item I’d like to explain is the current delay.
                  To understand it you need to read the “How to Observe – Right
                  Ascension” section. As it is explained, the theory of right
                  ascension (RA) considers the value RA on the vernal equinox
                  (21st March). Progressively with the changing seasons this
                  time varies, although not the actual RA of the star but the
                  delay it has on the Sun. Why? Because while the Sun increases
                  the time of daylight and its height above the horizon, stars
                  do not, not in human timespan at least. Considering a full 24
                  hours to go back to their original RA and 12 months to do so,
                  every month the RA move back 2 hours. Again, I encourage you
                  to read the dedicated section that further elaborate on this
                  aspect.
                

              

              


              

              

                
Closest Stars

                

                /astropy/api/v1/star/closest/<int>


                

                  This route accepts GET requests where only one argument is
                  required, namely the appid. The <int> in the URL
                  indicates the limit of results this query will yield. There
                  needs to be a limit. If it is below 0, default will return 5
                  results. On the other hand, the maximum number of results is
                  25. 


                  As the name suggests, if the request is successful, the return
                  value will be a JSON response with a number of stars specified
                  in the URL ordered by distance. This gives you a list of the
                  closest stars to Earth.
                  


                  In case the URL is incorrectly prompted a 404 Not Found
                  response will be received.
                

              

              


              

              

                
Brightest Stars

                

                /astropy/api/v1/star/brightest/<int>


                

                  This route work exactly as the previous one, but the return
                  JSON response will see the stars ordered by their apparent
                  magnitude (brightness). Keep in mind that the first result
                  will have the lowest apparent magnitude because, as already
                  mentioned, the lower the magnitude the higher the brightness.
                  It could also go below 0, as a matter of fact it does.
